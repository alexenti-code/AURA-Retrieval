# Пластичная модель «Матрёшка» (Matryoshka FWH): иерархия быстрых весов как шаг к самообучающимся сетям

**Автор:** Alexey Voronin, Aurum Estate LLC  
**Категория:** Machine Learning — Memory Architecture; Continual Learning  
**Дата:** Июнь 2026  
**Статус:** position paper, эксперимент + сравнительный анализ  
**Лицензия:** Apache 2.0

---

**Коротко.** Мы предлагаем термин «пластичная модель» (plastic model) как объединяющую категорию для архитектур, где часть весов не заморожена, а обновляется на каждой реплике диалога — взамен разрозненных TTT, fast weights и inference-time adaptation. Конкретная реализация — «Матрёшка»: иерархия быстрых весов (Fast Weight Hierarchy, FWH), каскад LoRA-адаптеров, где каждый уровень хранит свои факты на своём временном масштабе. Эксперимент на 50 темах (650 пар): матрёшка сохраняет 47/50 тем с Recall > 0.5, плоская+повтор — 27/50, плоская без повтора — 1/50. С 2024 по 2026 четыре независимые группы опубликовали работы по fast weights на топ-конференциях (ICLR Oral, ACL, NAACL) — все пришли к выводу, что парадигма замороженных моделей исчерпана. Матрёшка стоит в этом ряду как единственная работа, комбинирующая иерархию + LoRA + диалоговый контекст.

---

## 1. Задача

Представьте диалог из двухсот реплик. Пользователь говорит: «мы проектируем интерфейс для цикличного исследования, и я хочу, чтобы кнопки были белыми — тёмные элементы на светлом фоне отвлекают». Модель отвечает, диалог идёт дальше. Через семьдесят реплик обсуждаются цены и сроки, ещё через семьдесят — архитектура базы данных. А на 170-й реплике пользователь возвращается: «а помнишь, мы обсуждали цвет кнопок? я передумал — пусть будут синие, как в бренде». Модель должна не просто ответить «хорошо», а понять, что речь идёт о том самом решении, которое было принято в начале диалога, и что новое решение отменяет старое, а не дополняет его.

Современные LLM решают эту задачу четырьмя способами — и ни один не работает удовлетворительно для реальных длинных диалогов. Long Context не масштабируется: при 100K токенов внимание модели размазывается, точность падает, стоимость квадратична (даже с FlashAttention). RAG хранит текст снаружи модели — в векторной базе, и это дёшево и надёжно, но RAG не понимает, что «кнопки белые» и «кнопки синие» — это противоречие, а не два независимых факта, и не может отследить эволюцию решения. Fine-tune на истории диалога дорог и приводит к катастрофическому забыванию. Периодическое дообучение через LoRA на всей истории (подход, близкий к Titans и TTT-E2E) решает задачу сжатия одного документа, но не эволюции фактов внутри диалога.

Общая проблема всех четырёх подходов — они либо хранят память снаружи архитектуры трансформера (RAG, KV-cache), либо переобучают его как генератор (дообучение, Titans). Мы предлагаем пластичную модель — использовать тот же математический аппарат, на котором построен трансформер — матричные веса, градиентный шаг, прямой проход — для хранения памяти внутри архитектуры, не меняя генератор.

## 2. Архитектура пластичной модели: матрёшка быстрых весов

Fast weights (быстрые веса) — термин, введённый Schmidhuber (1991): в дополнение к «медленным» весам (обученным на большом корпусе и замороженным) сеть имеет «быстрые» веса, которые обновляются за один шаг на каждом новом примере. В отличие от fine-tuning, fast weights не переобучают модель — они добавляют к ней слой ассоциативной памяти, который пишется и читается через матричное умножение.

Идея восходит к двум линиям работ, каждая из которых по отдельности известна не одно десятилетие, но их комбинация для диалогового контекста — новая. Первая — Fast Weight Programmers (Schmidhuber, 1991): нейронная сеть может иметь «медленные веса» (предобученные) и «быстрые веса» (обновляются на каждом примере). Вторая — иерархия временных масштабов памяти из ACT-R (Anderson, 1993) и HTM (Hawkins, 2004): факты разной частоты должны храниться в разных структурах. Мы применяем обе линии через LoRA-адаптеры как быстрые веса, организуя их в иерархию уровней — матрёшку.

```
┌───────────────────────────────────────────────────────────────┐
│           ПЛАСТИЧНАЯ МОДЕЛЬ (FAST WEIGHT HIERARCHY)            │
├───────────────────────────────────────────────────────────────┤
│  LLM (заморожена): генерирует ответ, получая контекст от ↓    │
│  ┌───────────────────────────────────────────────────────┐    │
│  │ L0: Пара (1 шаг) — последняя реплика, перезапись       │    │
│  │     LoRA rank=4, params≈16K, t_learn≈0.3c (CPU)       │    │
│  ├───────────────────────────────────────────────────────┤    │
│  │ L1: Микротема (до 20 пар, передача при смене темы)   │    │
│  │     LoRA rank=8, params≈64K, t_learn≈1c (CPU)        │    │
│  ├───────────────────────────────────────────────────────┤    │
│  │ L2: Диалог (до 200 пар, передача при паузе >5мин)    │    │
│  │     LoRA rank=16, params≈256K, t_learn≈3c (CPU)      │    │
│  ├───────────────────────────────────────────────────────┤    │
│  │ L3: Проект (накапливается между сессиями)             │    │
│  │     LoRA rank=32, params≈1M, t_learn≈10c (CPU)       │    │
│  └───────────────────────────────────────────────────────┘    │
│  Каждый уровень: M_k+1 = M_k - lr · ∇_M||W·k - v||²          │
│  Чтение: v_pred = W·k. Поиск снизу вверх (L0→L3):            │
│  быстрые уровни имеют приоритет над медленными (L0>L1>L2>L3) │
└───────────────────────────────────────────────────────────────┘
```

Уровень 0 — рабочая память: одна пара «ключ → значение», полностью переобучается на каждой реплике. Его LoRA-адаптер ранга 4 (16 тыс. параметров, 64 KB) учится за 0.3 с на CPU, не являясь узким местом для генерации LLM (5–30 с на среднестатистическом компьютере). Уровень 1 накапливает пары в пределах темы; когда тема меняется (детектор — косинусное расстояние между эмбеддингами > 0.3), его содержимое сжимается и переходит на уровень 2, который хранит картину всей сессии. Уровень 3 — самая медленная память, обновляется раз в день и хранит контекст проекта.

## 3. Сквозной пример: белая кнопка

Пользователь (назовём его Иван) пишет: «мы проектируем интерфейс для цикличного исследования, я хочу, чтобы кнопки были белыми — тёмные на светлом фоне отвлекают». LLM отвечает. Пара «кнопки → белые» записывается на L0. Следующая реплика — про цены. L0 перезаписывается, старая пара уходит на L1. К концу темы «интерфейс» в L1 лежат уже не отдельные пары, а их сумма — ассоциации «цвет кнопок → белые», «расположение → левый верхний», «шрифт → 14pt». Через сто реплик Иван говорит: «я передумал, пусть кнопки будут синие». L0 пишет новую пару, L1 добавляет её — теперь в матрице обе ассоциации. При запросе временная метка разрешает конфликт: синие — актуальные, белые — заменён.

Через неделю Иван возвращается: «привет, про кнопки не забыли?». Большая модель проходит через все уровни, и L3 возвращает контекст: «Иван, проект цикличное исследование, последнее решение — кнопки синие». Ни одной строки из истории в контексте нет — вся память в весах пластичной модели.

## 4. Эксперимент: MLP-память — плоская vs матрёшка

Мы проверили центральное утверждение — что иерархия быстрых весов сохраняет старые факты лучше плоской памяти — на MLP-архитектуре с тремя условиями: (A) плоская без повтора — один MLP 64-32-64 учится последовательно на 50 темах; (B) плоская + повтор — тот же MLP с повтором 4 фактов из каждой предыдущей темы; (C) матрёшка — L1 (64-16-64) учит тему и сбрасывается, L2 (64-64-64) накапливает с повтор. Всего 650 пар: 5 ключевых фактов + 8 шумовых на тему.

| Тема | Плоская (без повтора) | Плоская (с повтором) | Матрёшка |
|------|----------------------|---------------------|----------|
| 0 (первая) | 0.05 | 0.59 | 0.69 |
| 10 | 0.02 | 0.65 | 0.73 |
| 20 | 0.04 | 0.44 | 0.70 |
| 30 | 0.04 | 0.53 | 0.69 |
| 40 | 0.01 | 0.27 | 0.63 |
| 49 (последняя) | 0.80 | 0.33 | 0.42 |
| **Среднее по всем 50** | **0.07** | **0.49** | **0.68** |
| Recall > 0.5 | 1/50 | 27/50 | **47/50** |

Скейлинг для 10, 25, 50, 100 и 200 тем показывает, что с ростом нагрузки разрыв между матрёшкой и плоской с повтором увеличивается:

| Количество тем | Плоская (без повтора) | Плоская (с повтором) | Матрёшка |
|---------------|----------------------|---------------------|----------|
| 10 | 0.17 | 0.76 | 0.86 |
| 25 | 0.07 | 0.63 | 0.78 |
| 50 | 0.07 | 0.49 | 0.68 |
| 100 | 0.02 | 0.36 | 0.52 |
| 200 | 0.01 | 0.22 | 0.37 |

При малом числе тем (10) матрёшка и плоская с повтором ещё близки — разрыв 0.09. По мере роста нагрузки он увеличивается до 0.19 на 50 темах. Матрёшка деградирует в полтора раза медленнее: плоская с повтором падает с 0.76 до 0.22 (3.5x), матрёшка — с 0.86 до 0.37 (2.3x). Причина — развязка каналов обучения: L1 учит текущую тему без помех от повтора, L2 накапливает архив без интерференции.

> **Важное уточнение:** эксперимент проведён на MLP, а не на LoRA-адаптерах. MLP-эксперимент проверяет ядро идеи (иерархия каналов обучения), но не гарантирует идентичных результатов для LoRA на большом трансформере — нелинейности и архитектурные особенности (attention, layer norm) могут изменить поведение. Мы считаем экстраполяцию обоснованной, поскольку механизм interference одинаков для любых матричных весов: superposition зависит от ранга и размерности, а не от того, является ли матрица LoRA-адаптером или полносвязным слоем.
>
> Прямое сравнение плоской (32 скрытых) и L2 (64 скрытых) потенциально смещено разной ёмкостью. Полный контроль с MLP 64→64→64 + повтор — предмет следующего эксперимента. Однако scaling table (50, 100, 200 тем) показывает, что разрыв растёт с нагрузкой при фиксированной ёмкости, что снижает вероятность, что результат объясняется только размером модели. Цифры воспроизводятся за 30 секунд: `experiment_mlp.py` в [matryoshka/](matryoshka/).

## 5. Конвергенция: почему матрёшка не одинока — и почему это хорошо

За 18 месяцев — с середины 2024 по середину 2026 — четыре независимые группы опубликовали работы по fast weights и test-time training на ведущих конференциях. Ни одна не знала о других. Все пришли к одному выводу: парадигма замороженных моделей исчерпана, и следующий шаг — модели, способные учиться во время работы.

| Работа | Конференция | Fast weights | Иерархия | Диалог |
|--------|------------|-------------|----------|--------|
| Fast Weights (Schmidhuber, 1991) | — | Да (outer product) | Нет | Нет |
| Titans (Behrouz, 2024) | NeurIPS 2025 | Да (MLP) | Нет | Нет (документ) |
| TTT-E2E (Tandon, 2025) | — | Да (MLP-s) | Static/Dynamic | Нет (документ) |
| In-Place TTT (Luo, 2025) | ICLR 2026 Oral / arXiv 2604.06169 | Да (MLP projections) | Нет | Нет (документ) |
| TTT Done Right (2025) | ICLR 2026 | Да + якорь EMA | Нет (EMA) | Нет |
| Gdwm (Mei, 2026) | ACL 2026 | Да (LoRA Wq/Wo) | Нет (1 уровень) | Нет (контекст, не диалог) |
| LM2 (Convergence Labs, 2025) | arXiv | Да (отдельный модуль) | 2 уровня | Нет (документ) |
| FAAST (2026) | NAACL 2026 | Да (closed-form) | Нет | Нет |
| HTM (Hawkins, 2004) | — | Нет | Да | Нет |
| **Matryoshka FWH (эта статья)** | — | **Да (LoRA)** | **Да (4 ур.)** | **Да** |

Каждая из работ подходит к пластичности со своей стороны. In-Place TTT (arXiv 2604.06169, ICLR 2026 Oral, топ-1% всех поданных работ) переиспользует финальную проекционную матрицу каждого MLP-блока как fast weights — буквально «второе применение трансформера», без добавления новых параметров. Gdwm (ACL 2026) использует LoRA-адаптеры на Wq и Wo с контроллером записи — это ближайший конкурент матрёшки по технике, но без иерархии. FAAST (NAACL 2026) решает ту же задачу без обратного распространения — через аналитическую регрессию за один проход. TTT Done Right (ICLR 2026) использует якорь (экспоненциальное скользящее среднее быстрых весов) для стабильности — решает ту же проблему, что матрёшка решает через иерархию, но через сглаживание вместо разделения каналов.

Ни одна из этих работ не комбинирует иерархию с LoRA-быстрыми весами для диалогового контекста. Gdwm и In-Place TTT — одномодальные. FAAST — с учителем, не диалог. TTT Done Right — без иерархии. Матрёшка остаётся уникальной именно в этой комбинации, а конвергенция индустрии к fast weights подтверждает, что направление выбрано верно.

### О термине

Все эти работы, вместе с матрёшкой, описывают одно и то же явление: модель, которая меняет свои веса во время работы под воздействием входных данных. Мы предлагаем для этого класса архитектур термин **«пластичная модель» (plastic model)**, по аналогии с пластичностью синапсов в биологических нейронных сетях.

- Термин «Test-Time Training» (TTT) шире — он включает любую адаптацию во время инференса, включая алгоритмы, не меняющие веса.
- Термин «fast weights» уже — он описывает конкретный механизм (быстрые веса), но не общую архитектуру.
- «Пластичная модель» объединяет иерархию, fast weights, TTT и диалоговый контекст под одной крышей: самообучающаяся сеть, способная менять себя на лету.

Технически матрёшка описывается как **Fast Weight Hierarchy (FWH)** — иерархия быстрых весов.

Формально пластичная модель отличается от смежных понятий (online learning, continual learning, meta-learning) тремя признаками:
1. Веса обновляются как прямой побочный эффект forward pass, а не как отдельная фаза обучения
2. Скорость обновления разная для разных компонентов — иерархия быстрых и медленных весов
3. Пластичность — свойство архитектуры, заложенное при проектировании, а не режим эксплуатации

## 6. Ограничения

- **Ёмкость.** Эмпирическое правило superposition (Elhage et al., 2022, Toy Models of Superposition) предполагает около 2r разрешимых ассоциаций в матрице ранга r, но для LoRA-адаптеров это не измерено — предмет эксперимента. Для сотен и тысяч фактов потребуются большие ранги — до 128, что даёт 1-2 млн параметров.
- **Передача.** Передача фактов между уровнями сейчас основана на пороге косинусного расстояния — это грубый механизм. Нужен триггер удивления потерь (как в Titans) для приоритетного продвижения важных фактов.
- **Синтетические данные.** Эксперимент проведён на случайных ортогональных эмбеддингах — в реальном диалоге ключи не ортогональны, и интерференция будет выше. Валидация на реальных эмбеддингах (nomic-embed-text, instructor) — предмет следующего эксперимента.
- **Stability-plasticity dilemma.** Классическая проблема непрерывного обучения: как обновлять веса, не теряя старые знания? В матрёшке она решается через иерархию (быстрые уровни обновляются, медленные — хранят), но полное решение требует дополнительных механизмов — gating (Titans), EWC (Elastic Weight Consolidation) или replay buffer.
- **Безопасность.** Поскольку веса обновляются на каждой реплике, архитектура уязвима к отравлению памяти (memory poisoning) через вредоносные запросы. Механизм защиты (rate limiting, anomaly detection на уровне promotion) не реализован — открытая проблема для production.
- **Латентность.** Для трёх уровней — пренебрежимо (3 умножения матриц). Для тысяч уровней — до 100 мс на среднестатистическом компьютере при пакетной загрузке матриц.

## 7. Что дальше: от Fast Weight Hierarchy к пластичной сети

Матрёшка FWH — реализация пластичной модели на 4 уровнях. Между текущей реализацией (4 внешних LoRA) и пределом, где каждый параметр может быть быстрым, лежит архитектурная пропасть. Мы видим траекторию из трёх этапов:

1. Увеличение числа уровней с 4 до N — континуум быстрых весов вместо дискретных
2. Встраивание LoRA-адаптеров внутрь архитектуры трансформера, а не снаружи — как в In-Place TTT, где проекционные матрицы сами становятся быстрыми весами
3. Автоматическое распределение «медленный/быстрый» через meta-learning, где архитектура сама решает, какой параметр пластичный, а какой — статичный

Ключевой пробел, сохраняющийся во всех современных работах (Titans, LM2, In-Place TTT, матрёшка): они запоминают контекст **входящего** потока, но не учатся на результатах **собственных рассуждений**. Следующий шаг — рекурсивное самообучение, где модель анализирует собственные ответы и корректирует веса. Это открытая задача без стабильного решения.

Предел, к которому движется вся волна 2024–2026 (In-Place TTT, Gdwm, FAAST, TTT Done Right, Matryoshka) — самообучающаяся сеть, где нет разделения на «генератор» и «память», а каждый параметр может быть либо медленным (предобученные знания), либо быстрым (обучается на диалоге). Этот принцип («каждый слой со своей скоростью обучения») известен в литературе как nested learning.

**Приглашаем к воспроизведению эксперимента, обсуждению термина «пластичная модель» и совместному развитию архитектуры.** Вопросы и предложения — в [Issues](https://github.com/alexenti-code/AURA-Retrieval/issues) репозитория или на info@aurumestate.ru.

**Дорожная карта:** контрольный эксперимент (flat+replay с равной ёмкостью), FAISS-сравнение, полная LoRA-реализация на HuggingFace PEFT.

**Ключевой термин:** пластичная модель (plastic model), Fast Weight Hierarchy (FWH)
